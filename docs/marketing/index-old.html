02 · the problem

Every generative UI demo is missing the same thing.

Look at every viral AI-generated interface of the last two years. The model emits markup. The runtime renders it. The user sees it. Then ask one quiet question.

Where is the plan?

There isn't one. Nothing in between catches a mistake. Nothing anchors identity. Nothing survives a refresh. Nothing diffs. Nothing replays. The intelligence and the rendering live in the same step, and you cannot pry them apart.

Axiom 01
Generative UI is a parlor trick.
Selective UI is a product.
Selective UI
The model doesn't generate your interface. It picks one from a finite, typed grammar of what your application has declared it can do. A parser into your vocabulary, not an author of new ones.

Six phases in the database. Six phases here. Four of ours are glass — typed, deterministic, debuggable. Two are opaque — and they're collapsed into a single model call, the way an optimizer is a single opaque box in a query planner.

04 · intelligence

The model is the cost-based optimizer.

In SQL, the optimizer is the smart, opaque box. Statistics, heuristics, dynamic programming — picking a plan from a search space against an internal cost function.

In Resonance, the language model plays that role. Same job, different substrate. It selects a plan from a search space using learned representation in place of histograms, and a semantic cost function in place of latency. The mechanism is opaque. The role is identical.

Then the model gets out of the way. The plan is typed. The compiler is total over valid plans. The renderer is Phoenix. Everything downstream of the optimizer is glass.

06 · the proposal

We propose three pieces. We call them MPC.

One piece the application declares, one the model emits, one the runtime runs.

M
Manifest
— what the app can do

Datasets, dimensions, measures, patterns, primitives.

P
Plan
— what should appear

A typed, validated value the model emits.

C
Compile
— how it becomes a surface

Plan in, Renderables out. Deterministic.

07 · manifest
M
Manifest
— the catalog, for agents

The Manifest is what your application has declared it can do. It's the agent-readable interface to a real product — the database's information_schema, lifted to product capabilities.

A Resonance app names its datasets (deals, customers, orders), the dimensions and measures each accepts, the filters they understand, and the semantic primitives the model is allowed to use over them — rank, compare, distribute, segment, summarize. Eventually, actions.

Plans bind against this manifest. A plan that names a dataset the app never declared fails at validation, before anything renders. The trust boundary is the type system.

manifest.ex  ·  CRMResolver.describe/0
%{ datasets: [ "deals" => %{ dimensions: [:stage, :owner, :closed_at, :region], measures: [:count, :sum_value, :avg_value], filters: [:closed_at, :owner, :stage] } ], primitives: [:rank_entities, :segment_population, :compare_over_time, :summarize_findings], patterns: [:bar_chart, :trend_panel, :entity_list, :data_table, :prose_summary] }
08 · plan
P
Plan
— the artifact

The Plan is a typed, validated value the model emits. A workspace is a goal, a layout, and a list of sections. Each section names a role, a presentation pattern, and a source — a typed tool call that returns a Result.

The plan is a value. The rendered page is one expression of it. The saved JSON snapshot is another. The same plan replayed against tomorrow's data, with no new model call, is a third.

workspace_plan.ex  ·  emitted by the optimizer
%WorkspacePlan{ goal: :pipeline_review, layout: :overview_with_detail, identity: "crm:pipeline-review", sections: [ %Section{ id: "summary", role: :summary, pattern: :prose_summary, source: {:tool_call, %ToolCall{...}} }, % … 4 more sections … ] }
09 · compile
C
Compile
— plan to surface

The Compiler takes a Plan in and produces Renderables out. It walks each section, resolves the source through your Resolver, hands the Result to a Presenter, stamps a stable identity, and produces a list of components Phoenix will mount.

The compiler is total over valid plans. It can only fail on invalid plans — and validation catches those before compilation begins. Per-section failures become explicit error Renderables. Other sections render anyway.

Resonance composes what should appear. Phoenix runs what appears.
