# Resonance v3 Execution Plan: Adaptive Workspaces

> Status: draft execution plan. This translates
> `docs/design/v3-composition-vision.md` into a staged build plan. It is not
> yet a ticket list, but it is shaped so each phase can become tickets.

## The Recommendation

Build v3 through a v2.5 middle layer: a first-class workspace plan that can be
hand-authored, validated, compiled into today's Renderables and Widgets, saved,
and rerun before an LLM ever emits one.

The planner is not the hard center. The compiler is.

If a hand-written plan cannot become a coherent LiveView workspace with stable
section identity, refresh behavior, and understandable failures, an LLM planner
will only make the failure harder to debug. Once the deterministic compiler
works, the LLM planner has a real target.

## Non-Negotiables

- No LLM-emitted HEEx.
- No raw atom tree as the primary IR.
- No Resonance-owned app data layer.
- No Resonance-owned mutation runtime.
- Phoenix remains the runtime: state, events, PubSub, forms, and validation stay
  in the consuming app.
- Resonance composes what should appear; Phoenix runs what appears.

## Current Leverage

The repo already has the right lower layers:

- `Resonance.Pipeline.run/3` is the single streaming path from prompt to events.
- `Resonance.Pipeline.resolve/3` can rerun stored tool calls without an LLM call.
- `Resonance.Result` separates semantic truth from presentation.
- `Resonance.Presenter` maps truth into developer-owned UI.
- `Resonance.Widget` proves interactive LiveComponents can be composed without
  Resonance owning their runtime behavior.

The missing layer is not "more widgets." It is a workspace IR above tool calls
and below rendered LiveView.

## Phase 0: Stabilize the Ground

Goal: remove ambiguity before adding a planner.

Work:

- Resolve the stale v2 docs/runtime contract around widgets and PubSub.
  Parent LiveViews should own subscriptions and forward updates; widgets should
  own local interaction and rendering.
- Finish or fold in the failure policy work from TK-048.
- Land structured resolver description work from TK-036/TK-059.
- Land Result format metadata from TK-038.
- Decide whether TK-057's LLM retry feedback loop is a dependency for planner
  mode or can wait until after the deterministic compiler.
- Update stale project docs that still describe the old test count or pre-v2
  architecture.

Acceptance:

- Existing `mix test.all` passes.
- Widget docs, examples, and code say the same thing about runtime ownership.
- Known failure modes have explicit user-facing behavior and telemetry/logging
  expectations.

## Phase 1: Workspace IR and Deterministic Compiler

Goal: prove that a typed workspace plan can compile into today's rendering
system without involving LLM planning.

Add:

- `Resonance.WorkspacePlan`
- `Resonance.WorkspacePlan.Section`
- `Resonance.WorkspacePlan.Validation`
- `Resonance.WorkspaceCompiler`

Initial plan shape:

```elixir
%Resonance.WorkspacePlan{
  goal: :pipeline_review,
  title: "Pipeline review for this week",
  layout: :overview_with_detail,
  sections: [
    %Section{
      id: "summary",
      role: :summary,
      pattern: :prose_summary,
      source: {:tool_call, %Resonance.LLM.ToolCall{...}}
    },
    %Section{
      id: "stuck_deals",
      role: :focus_list,
      pattern: :entity_list,
      source: {:tool_call, %Resonance.LLM.ToolCall{...}},
      interactions: [:filter, :inspect]
    }
  ],
  refinements: [],
  identity: %{kind: :ephemeral, saveable: true}
}
```

Keep the first schema small:

- Layouts: `:stack`, `:dashboard_grid`, `:overview_with_detail`
- Roles: `:summary`, `:primary`, `:focus_list`, `:supporting_context`,
  `:detail`
- Patterns: start as named wrappers over existing presenter outputs, not a new
  design-system layer.
- Sources: only `{:tool_call, ToolCall.t()}` in phase 1.

Acceptance:

- A hand-written CRM workspace plan validates.
- The compiler resolves each section through the existing pipeline/composer
  path.
- Section IDs produce stable Renderable IDs.
- `Live.Report` remains compatible with v1/v2 reports.
- Tests cover validation, compile success, compile failure, and stable IDs.

## Phase 2: Workspace Identity, Save, and Rerun

Goal: make generated workspaces revisitable before making them generated by an
LLM.

Add:

- A serializable workspace plan representation.
- A deterministic plan fingerprint.
- `Resonance.WorkspaceSnapshot` or equivalent value object containing the plan,
  resolved section metadata, original prompt, and created-at timestamp.
- A rerun path that uses stored section sources and `Pipeline.resolve/3`.

Resonance should not own persistence. The consuming app stores the serialized
snapshot wherever it wants. The CRM demo can store in a minimal app-owned table
or in-session store for proving the loop.

Acceptance:

- Save a hand-written workspace.
- Change CRM data.
- Rerun the workspace without a new LLM call.
- Existing section IDs update in place.
- The workspace still reads as the same workspace after data changes.

## Phase 3: Planner Prototype

Goal: let the LLM emit a workspace plan only after the compiler has a firm
contract.

Add:

- `Resonance.Planner.plan/3`
- A single planner tool/schema such as `create_workspace_plan`.
- Plan validation with actionable errors.
- Optional one-shot retry using validation feedback.
- A golden prompt evaluation harness for CRM questions.

The planner should emit workspace structure and section sources. It should not
emit HEEx, component modules, CSS classes, or raw component trees.

Acceptance:

- Five to ten CRM prompts produce valid plans.
- Every valid plan compiles.
- Invalid plans return structured validation errors.
- One validation-feedback retry materially improves success rate.
- Planner tests use mocked providers at the provider boundary, not mocked
  internals.

Target bar:

- 70% valid without retry on the first prompt set.
- 95% valid with one retry before considering planner mode real enough to
  dogfood.

## Phase 4: Capability Manifests and Pattern Kit

Goal: give the planner richer material than primitive tool names while staying
above raw UI atoms.

Add:

- Data capability manifests from structured resolver `describe/0`.
- Interaction capability manifests from widgets and patterns.
- `Resonance.Pattern` behavior, if the shape earns it.

Possible pattern behavior:

```elixir
@callback describe() :: map()
@callback accepts?(Resonance.WorkspacePlan.Section.t(), map()) :: boolean()
@callback compile(Resonance.WorkspacePlan.Section.t(), map()) ::
  {:ok, Resonance.Renderable.t()} | {:error, term()}
```

Start with a tiny pattern kit:

- `:metric_strip`
- `:entity_list`
- `:trend_panel`
- `:summary_panel`
- `:comparison_panel`
- `:data_table`

Acceptance:

- Planner sees patterns and capabilities, not component modules.
- Validation catches incompatible role/pattern/source combinations.
- The CRM demo can declare product-specific patterns without forking
  Resonance internals.

## Phase 5: Live Workspace Surface

Goal: expose adaptive workspaces as a real product surface rather than a report
with a better struct behind it.

Add:

- `Resonance.Live.Workspace` or a clearly separated workspace mode.
- Prompt-to-plan streaming states: planning, resolving, ready, failed.
- Save, rerun, and refine affordances.
- Workspace-scoped conversation context, likely absorbing TK-037.
- Section-aware refinements, such as "show only negotiation deals" against a
  focus list section.

Do not overload `Live.Report` until the workspace lifecycle is proven. It can
share internals, but the surface contract is different enough to earn its own
module.

Acceptance:

- A user can generate, save, rerun, and refine a CRM workspace.
- A saved workspace survives a LiveView reconnect.
- A second prompt while one is running has defined behavior.
- Planner and resolution telemetry make latency visible.

## Phase 6: Action Surfaces

Goal: test whether workspaces can safely compose "do" surfaces, not just "show"
surfaces.

Defer this until read-only/refine-only workspaces are credible.

Add one safe CRM action first:

- Action manifest: name, description, parameter schema, authorization notes.
- Developer-owned validator.
- Developer-owned handler.
- Generated form surface with user confirmation.

Rules:

- The model may suggest and prefill.
- The user confirms.
- The app validates and performs.
- Resonance never bypasses app-owned trust boundaries.

Acceptance:

- One action surface can be planned, rendered, submitted, rejected on invalid
  input, and performed on valid input.
- All mutation behavior is app-owned and testable without an LLM call.

## Deferrals

Do not build these in the first serious version:

- Raw design-system atom composition.
- Arbitrary Phoenix component selection by the model.
- LLM-generated copy for every section beyond title/intent labels.
- Multi-user collaborative workspace editing.
- Cross-app persistence adapters.
- Full action composition.
- Automatic promotion of workspaces into nav/pages.

They may become real later. Building them now would widen the blast radius
before the core mechanism proves itself.

## Success Criteria

Technical:

- Existing v1/v2 APIs keep working.
- Workspace plans validate before resolution.
- Every planner output either compiles or returns actionable validation errors.
- Stable IDs survive reruns.
- Latency is measured by phase: LLM planning, section resolution, render.

Product:

- Users save or rerun workspaces because they remain useful after the first
  prompt.
- Users refine sections rather than regenerate from scratch every time.
- At least some useful workspaces are narrow enough that a team would not have
  pre-built them as fixed pages.

Kill criteria:

- Hand-written plans do not compile cleanly into coherent workspaces.
- Saved plans do not feel stable after rerun.
- Planner validity remains poor after validation feedback.
- Latency makes the surface feel like a worse dashboard.
- Users do not return to generated workspaces.

## The Caveat

This plan is less flashy than jumping straight to "the model builds an app
screen." Good. Flash is cheap here. The thing worth proving is whether a user's
intent can project a durable, app-native workspace from declared capabilities.
That proof starts with a deterministic compiler, not with more model freedom.
